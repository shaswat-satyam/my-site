  2. Repo

    Repo or Repository is nothing but a folder which we want to track git is an acyclic graph meaning it cannot contain loop Each commit is a node in the graph and each pointer is the child to parent relationship Also known as work tree git --version - TO view the git version the system git status - To view the current status of the repo in the current folder git init - To start a repository in the current folder. It creates .git folder

    .git is a hidden folder to store the various commits, hooks and code snapshots To remove a repository delete this .git folder

    Commit are like checkpoints . We write some code. then add it to the git staging area then commit it to the repos It produces a unique SHA for identification It represents the project in its entirety

    Working Directory -> git add -> Staging area -> git commit -> Repo -> git push -> Github

1.3. Staging area

staging area can be also referred as index pwd to print current directory we work on some branch git lens to watch the git commit tree touch one.txt two.txt git status - Above two files go into untracked files git add . Adds all files to tracking git add one.txt Add only one.txt to tracking git rm --cached file.txt Remove file from staging area git commit -m "Added first files" Commit the current staging area with the message Without the -m file opens the default editor to ask for message By default It opens vim By Opnion Commit messages should be present tense and imperative git log Shows the log file for the git history It shows the commit with SHA hash with branch the author, email and TimeStamp git log --oneline - Short form of log each commit in a single line

1.4. Config File

Git maintains a config file to manage various important details about the system and the user like email or name git config --global core.editor "code --wait" git config --global user.email "email@mail.com" git config --global user.name "username"

.gitignore is a file containing the list of files to ignore in the repository like environment variables API keys or other sensitive information It can contains the name of the files and directory as it is COMMITEDITMSG is edited everytime the commit message is written It contains git log as comments .gitignore file can be generated by gitignore generator

config file is stored in the root of system as .gitconfig it is global config file

Commit is dependent on the previous commit except the first commit Each commit contains a pointer to the previous commit

Commits, Head are stored in .git folder Hooks are used to execute scripts just before committing

Head is stored in .git/logs It stores the commit history

Branches are stored in .git/logs/refs

1.5. Branch and Merging

Branches are like various timelines so people can work on some code without interferring with other code git commit -m "Add index file" git branch Gives the current and all the other branchs By default branchs were named as master now they are named main

git switch main Alternative to git branch main git switch -c newBranch Creates a new branch and moves the head to it git checkout -b newBranch2 Creates a new branch and moves the head to it

Head contains the path to the current branch refs/heads/master

git branch main Creates a new branch main but not points to it git checkout main Moves the head to the main A branch can’t effect other branch until they are merged

The Head can also moves within the branch to previous commits

git merge TO merge various branches There are two types of merge fast forward merge and non fast forward merge In fast forward merge We create a new branch at the end of the main branch and then convert this new branch to main branch In non fast forward branch the simulatenous work is done on both main and new branch and if they are merge capable they are merged together or in short we resolve conflicts

first get the main branch using git checkout git merge nav-bar -m "nav-bar merged in main" It merges the nav-bar to the main branch as a commit with the commit message

git branch -d nav-bar it deletes the nav-bar branch in the present context

If there are conflicts Then the editor will open showing the code on the current branch Then the code on the other Branch seperated by equals Then we have to decide what to keep and what to discard Edit the file as needed and try to merge again

1.6. Git Diff and Stashing

git diff --staged Used to compare code with other commits, branch interactively for the staged file git diff CommitID CommitID2 Used to compare code between two commit IDS

git diff branchOne..branchTwo Used to compare code between two branches

  • and - doesn’t mean code wass added or removed they just represent the change between the two files

Git Stash is used to store current uncommitted code while you can do other work Stash can be moved from one branch to another Git Stash Stash the work Git Stash list List all the stash Git Stash pop Bring back changes Git Stash apply stash@{0} Apply the changes and then keep in the stash

git checkout hash Checkout a particular commit git switch main go the head of main branch Git checkeout head~2 go the second before the head of the current branch git restore file Restore the file to its latest commit git reflog got to the last position of head

1.7. Git Rebase

It is used to rewrite the history It is an alternative to mergin Never run this on the main branch git commit -am "commit message" It is a shortcut to add all the files and commit them On Merging we create a new commit To merge without that commit we can use rebase git rebase master To rebase the current branch with the master branch Timeline is rewritten and moved every commit on the current branch to the master branch Both branches point to the same branch For Conflicts Resolve all conflicts git add the files Then use git rebase –continue or git rebase –abort
