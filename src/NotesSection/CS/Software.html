Both processes aim to create high quality reliable products Conventional engineering focus on physical systems,while software engineering deals with intangible software systems

Both processes involve iterative design and prototyping Software engineering allows for greater flexibility and ease of modificiation due to non physical nature of software

Both processes emphasize the importance of testing and quality assurance to ensure product performance and reliability Conventional engineering often involves physical testing and prototypes while software engineering relies on various types of testing such as unit integration and system testing

1.9.1. Waterfall Model

Inspired by manufacturing and construction process where each step relies on the completion of the previous one It is a simplest SDLC in which phases are organized in a linear and sequantial order It is called waterfall model as it diagrammatic reprsent as like a water also known as classical life cycle model This type of model is basically used for the small to medium sized project with clear well defined requirements. When the technology and tools to be used are well known and stable Projects where minimal changes are expected during the development process and predictability is prioritized over adaptibiltiy

  1. Advantages

    Easy to understand and implement, with well defined stages and clear milestones Each phase has well defined input and output phases are porcessed and completed one at at time. Phases do not overlap Lowe cost and easy to schedule as all staff do not work concurrently on the same project, so can work on different projects.

  2. Disadvantages

    Not suitable to accommodate any change or iteration once development has begun. It is alway difficult to acquire of the requirement in the starting Working version is produced in the last level, so not good for large size sophisticated projects. High amount of risks and uncertanity.

1.9.2. Prototype model

It is an iterative approach, which involves developing an early working model of the software based on the currently known requirements with limited functionalites, low reliability and untrusted performance Refining through user feedback, and repeating the process until a satif then shown to the user as per the feedback fo the user prototype is rebuilt and modified and again shown to the user, the process continue till the customer is not satisfied. After this process the final SRS document is developed Developing the prototype helpp in building the actual design Prototype model is of two types

  1. Evolutionary prototype

    Also known as incremental or iterative prototyping, develops a working prototype gradually improving and refining it based on user feedback Prototype evolves into the final product over time with feature additions and modifications. Accommodates changing requirements during the development process.

  2. Throwaway prototype

    It is also known as rapid or exploratory prototyping, involves creating a temporary, simplified version of the software to validate the feasibility of specific features, test ideas or gain user feedback on the inital plam. The throwaway prototype is not intended to be used in the final product instead it is discarded once it has served its purpose. Once the developers have gathered the necessary information, They can then start developing the actual software product from scratch, using the lessons learned from the prototype. This approach is particularly useful for testing complex or risky features as it helps to minimize the potential impact of these feature inside the project.

  3. Advantage

    Cutomer get a chance to see the product early in the life cycle, and give important feedback. There is a scope to accommodate new requirements Developer and more confident, hence risk is reduced

  4. Disadvantage

    After seeing the early protoype use demand the actual system soon If not managed properly the iterative process can run for a long time If user is not satisifed, he may lose its interest in the project.

1.9.3. Spiral Model

The radial dimensions of the model represent the cumulatie costs. Each path around the spiral is the indicative of the increased cost. The angular dimenssions represent the progress made in the completing each cycle, each loop of the spiral from X axis clockwise through 360 represent one phase, which is divided into four sectors Determinig Obeject and plainning the next phase Risk Analysis Engineering the product, develop and test the product Customer evalutation

    Not suitable for small size project It is complex and time consuming to use Risk analysis require highly specific expertise

3.3. Types of Testing

3.3.1. Boundary Value testing

The black box testing techinques are helpful for detecting any errors or threats that happened at the boundary values of valid or invalid partitions rather than focusing on the center of the input data. We have a range for input values for the black test We use the boundary values

  1. Example

    Assume a test case that takes the speed of a car from 40 t0 80 to get the best fuel efficiency Test Cases Valid case - (Min,Min+1,Max-1, Max) INvalid case - (Min-1),(Max+1)

3.3.2. Equivalence Class testing

The input and output domain is partitioned into mutually exclusive parts called equivalence class Any one sample from a class is representative of entire class

  1. Steps
    1. Identify Equivalence classes by taking each input and output condition and partitioning it into valid and invalid classes
    2. Generate test cases using equaivalence classes.
  2. Example
    1. If a variable lies between 1 and 100 then equivalence classes will be Variable is below 1 Variable is between 1 and 100 included Variable is more than 100
    2. Previous Data program with inputs day month year O1 : All valid inputs O2: Invalid Date I1: 1 <= m <= 12 I2: m < 1 I3: m > 12 I4: 1 <= d <= 31 I5: d < 1 I6: d > 31 I7: 1900 <= Y <= 2025 I8: Y > 2025 I9: Y < 1900

3.3.3. Decision Table Based Testing

This testing technique is useful when a variety of actions exist and we need to take a particular action depending upon the exisiting condition If all entrires are binary then decision is called Limited Decision Table/ Limited Entry Decision Table If conditions can have several values then the decision table is called Extended Entry decision table Condition represent the input conditions that must hold Action represent the output conditions

  1. Parts of Decision Table
    1. Condition Stub
    2. Action Stub
    3. Condition Entries
    4. Action entrires
  2. Example
    T T T T F F F F
    T T F F T T F F
    T F T F T F T F

    Condition stub For C1, C2,C3 Merge the Trues and False consecutively

    X X     X     X
    X   X     X   X
      X     X      
          X   X    

    Action stub for a1,a2,a3,a3

    If C1 is true and C2 is true and C3 is true we get A1, A2 actions If C1 is true and C2 is true and C3 is false we get A1, A3 actions

3.3.4. Cause Effect Graph

It is a technique to systematically select a high yield set of test cases. Benefits - It points out the incompleteness and ambiguites in specifications

  1. Steps
    1. Identify the cause and effects from the specifications and assign unique numbers to each of them Cause - Any distinct input conditions or a equivalence class of input conditions Effect - Output condition or a system transformation (the effect that any IP has on the state of system )
    2. Draw a boolean graph linking causes and effects
    3. Specify constraints on graph describing combinations of cause and/or effects that are impossible
    4. Convert the graph into limited entry decision table by tracing state conditions in the graph Each column in DT is converted into a Test Case
  2. Notations
    1. Identify Function - Arrow with identify If C1 is 1, then e1 is also 1 else e1 is 0
    2. Not function - Arrow with AC current symbol If C1 is 1, then e1 is also 0 else e1 is 1
    3. OR function - Ci merge together with V IF any Ci is 1 then e1 is also 1 else 0
    4. AND function - Ci merge together with ^ If all Ci is 1 then e1 is also 1 else 0
  3. Example

    The characters in a column 1 must be A or B. The characters in column 2 must be a digit. If these 2 hold then fill update is made. If the character in column1 is incorrect, message x is issued. If character in column 2 is not a digit, message y is issued. Causes are -

    1. C1 - Character in column 1 is A
    2. C2 - Character in column 2 is B
    3. C3 - Character in Column 2 is a digit

    Effects are-

    1. e1 - Update made - (C1 V C2) ^ C3
    2. e2 - Message x is issued - ~C1 ^ ~C2
    3. e3 - message y is issued - ~C3

      1. Steps
        • Generating set of paths that will cover every branch in the program
        • Finding a set of test cases that will execute every path in this set of program paths
      2. Independent path

        Any path through decision to decision path graph that introduces atleast one new set of processing statements that introduces atleast one new set of processing statements or new conditions. It must move along atleast one edge that has not been traversed before

      3. Goals
        1. Every statement in the program has been executed atleast once
        2. Every branch has been exercised for true and false condition
    2. Data flow testing

      How data flows in system

      • Data flow testing focuses on two points
        1. In which statements the variables are defined
        2. In which statements the variables are used
      • It designs the test cases that cover control flow paths around variable definitions and their uses in the modules
      1. Example
        read a,b,c;
if(a>b)
  x = a+1;
  print x;
else
  x = b-1;
  print z;
        variables Define Use

      2. Advantage
        • A variable that is declared but never used within the program
        • A variable that is used but never declared.
        • A variable that is defined multiple times before it is used

3.3.6. Mutation Testing

  • It is a software testing where we mutate certain statement in the program and check if the test case are able to find the errors
  • The change in the mutant program are kept extremely small, so it does not effect the overall objective of the program
  • The goal of mutation testing is to develop effective test cases. It makes test cases robust
  • Also called fault based testing
  • Mutation score is the killed Mutant/ Total No of mutatants * 100 If we can develop test cases then mutation score will be 100%
  1. Execution

    Change the code and compare the output If the output is different then killed the mutant

  2. Example

    Original

    void calc(int n){
  if(n%2 == 0)
    printf("Even");
  else
    printf("Odd");
  return;
}

    Mutant

    void calc(int n){
  if(n/2 == 0)
    printf("Even");
  else
    printf("Odd");
  return;
}

    Results are differnent So we remove the mutant code IF the result would have been the same We would have changed the test cases.

  3. Types
    • Value mutation - We change the values
    • Decision Mutations - We change the decision statements
    • Statement Mutations - We change the statements
  4. Advantage
    • It brings a good level of error detection in the program
    • It discovers ambiguites in the source code.
  5. Disadvantage
    • It is highly costly and time consuming
    • It is not for black box testing

3.3.7. Unit testing

  • Unit tests are used to test individual blocks of functionality

3.3.8. Integration testing

  • It is conducted to evaluate the compliance of a system or component with specified functional requirements.
  • It occurs after unit testing and before system testing
  1. Types

      2 or 3 compatible units are combined first then testing begins Then integrate sequentially

    3. Top Down

    4. Bottom Up

3.3.9. System testing

  • It is a level of testing that validates the complete and fully integrated software product
  • The purpose of a system test is to evaluate the end to end system specifications
  • It is a black box testing
  • It based on categories : who is doing the testing
  • System testing ccategories based on : Functional/ Non - Functional requirements
  • Alpha testing is the checking by the creators
  • Beta testing is the checking done by the small group of users
  • Functional testing are the testing of the requirements mentioned on the SRS
  • Non Functional testing are the testing of the requirements not mentioned on the SRS, include load (upto the capacity), stress(outside the capacity).

3.6. Debugging

Identifying the cause of error. It is the activity or locating and correcting errors.

3.6.1. Characteristics

  • Symptom may be in one part cause might be in other part
  • Symptom may disapper when other error is corrected.
  • Symptom may be intermittent
  • Symptom might be due to human error and very difficult to trace.

3.6.2. Approach

  1. Print Statement
  2. Core Dumps
  3. Trial and Error
  4. Backtracking
  5. Commenting or Removing
  1. Induction and Deduction
    1. Keep the data handy All available data, symptoms about the problem, different test cases which make the error or symptom disappear.
    2. Organize the data Structure the data to observe patterns Search for specialized observations eg - Error only occurs when no stock is left
    3. Devise a hypothesis We have to theorize using all the data It is a capital mistak to theorize before one has data. we begin to twist facts to fit theories instead of twisting theories to fit facts.
    4. Prove the hypothesis Don’t skip this step to straightaway start fixing the problem Failure to do this might reduce only a portion of problem
  2. Deduction Approach
    1. Enumerate the possible causes Develop a list of all theories.
    2. Use the data to eliminate them If all are elminated,additional data is needed to devise new theories
    3. Refine the remaining hypothesis Use the available classes

    4. Prove the remaining hypothesis Enumerate possible causes -> Use Process of elimination -> Refine remaining hypothesis -> Prove remaining hypothesis

      Collect more data at every step

3.6.3. Tools

Debugging compilers, automatic test case generation, Memory dumps, Compiler diagnostics, compiler with meaningful error messages and error detection features

3.10.2. SCM

Software Configuration Mangement is defined as a process to systematically manage, Organize and control the changes in the doucments, codes and other entities during the SDLC The primary goal is to increase productivity with minimal mistakes.

  1. Need
    1. There are multiple people working on software which is continually updating
    2. It may be a case where multiple version, branches, authors are invloved in a software project, and the team is geographically distributed and works concurrently
    3. Changes in user requirement, policy, budget, schedule need to be accommodated.
    4. Software should be able to run on various machines and OS
    5. Helps to develop coordination among stakeholders
    6. SCM process is also beneficial to control the costs involved in making changes to a system
  2. Tasks
    • Software Configuration
    • Version Control Creating Versions of the exisiting product ot build new products from the help of SCM system VCS is a system or tool that captures the changes to a source code elements ie File, Folder, Image or binary
    • Change Control Change control is a systematic approach to managing all changes made to a product or system. The purpose is to ensure that no unnecessary changes are made that all changes are documented, that services are not unnecessarily disrupted and that resources are used efficiently
    • Configuration Control Software Configuration audits verify that all the software product satisfies the baseline needs. It ensures that what is build is what is delivered.
    • Status reporting Providing accurate status and current configuaration data to developers, tester, end users, customers and stakeholders through admin guides, user guides, FAQs, Release notes, memos, Installation Guide, Configuration guide
