1. Basics

1.1. OLAP vs OLTP

Aspect OLAP OLTP
Full From Online Analytical Processing Online Transaction Processing
Primary Function Designed for complex data analysis Handles daily transactional data processing
Database Design Star or snowflake schema optimized for read operations Normalized database optimized for write operation
Query Complexity Complex queries involving aggregations and computations across multiple dimensions Simple and standard queries focusing on CRUD operations
Data Volume Deals with large volume of data for historical analysis Processes a high number of small transactions
Respone time Slower response time due to complex queries Fast response time to support high transaction rate

1.2. Types of Database

1.2.1. Commerical Database

Predominately used in business sector to handle large volume of transactions and customer data.A CRM system like salesforce which handles large volume of customer data and transactions

1.2.2. Multimedia Database

Stores datatype such as images, audio and video files, facilitating the management and retrieval of multimedia content. A digital asset management system like adobe experience manager that facilitates the storage and retrieval of multimedia content

1.2.3. Deductive Database

Utilizes logic programming to derive information from data stored in a database, allowing for more complex and analytical queries. A database using Datalog which allows for complex logical queries and information derivation.

1.2.4. Temporal Database

Keeps track of changing data over time allowing for queries concerning time based data. A historical trading database in the final sector which keeps track of stock prices over time.

1.2.5. Geological Database

Stores, Organizes and analyzes geographical data, aiding in spatial analysis and mapping projects. A system ArcGIS which enables storage, analysis and visualization of geographical data.

1.3. Database Administrator

Database adminstrator hold authority over data and the program facilitating data access. Their roles/functions are:

1.3.1. Schema definition

DBA outlines the original database schema. Acheived through writing definition translated to permanent labels in the data dictionary by the DDL compiler

1.3.2. Storage structure and access methods Definition

Responsible for forming appropriate storage structures and access methods. Acheived through writing definition translated by the Data storage and definition language compiler.

1.3.3. Schema and physical Organisation Modifications

Involves altering the database schema or physical storage organisation. Changes are implemented by writing definition that modify the relevant internal system tables

1.3.4. Granting of authorization for data access

DBA grants varied type of data access authorization to different database users

1.3.5. Integrity Constraint Specification

DBA implements and maintains integrity constraint to maintain the accuracy and consistency

1.4. Database Architecture

1.4.1. Query Processor

It is the component of DBMS that interprets and executes user queries. It comprises several sub components including

  1. DML Compiler

    Process Data Manipulation Language statements into low level instructions that can be executed

  2. DDL Compiler

    Process Data definition Langauage statements into metadata tables

  3. Embedded DNL Pre Compiler

    Translates DML statements embedded in a application programs into procedural calls

  4. Query Optimizer

    Determines the most efficient way to execute a query by evaluating different query plans

1.4.2. Storage Manager

Also known as database control system. It is responsible for managging the data stored in the database, ensuring it consistency and integrity. It includes the following sub components

  1. Authorization Manager

    Manages access controls and privileges

  2. Integrity Manager

    Ensures the data modification adhere to integrity constraints

  3. Transaction Manager

    Manages concurrent access to the database and maintains database consistency during transaction

  4. File Manager

    Manages file space and data structure representing information in the database

  5. Buffer Manager

    Manages data cache and data transfer between main memory and secondary storage

1.4.3. Disk Storage

Represents the storage aspect of DBMS encompassing the followig components

  1. Data files

    Files where the actual data is stored

  2. Data dictionary

    Repository containing information about the structure and characteristics of database objects

  3. Indices

    Data structure that facilitates faster data retrieval

2. ER diagram

ER stands for entity relationship diagram This is central to database design, encapsulates entities and their attributes within an enterprise schema, serving clear and standardized tool for translating real enterprise interactions into conceptual schema

2.1. Entity

An entity is a thing or object in the real world that is distinguishable from other object based on the values of the attributes it possess. An entity can be concreter or abstract Can be

2.1.1. Tangible

Entities which physically exist in real world like Car Pen, locker

2.1.2. Intangible

Entities which exist logiically, Eg Account Video

We cannot represent an Entity in a ER Diagram. But a entity set

2.2. Entity set

Collection of same type of entites that share the same properties or attributes. In ER they are reprented by a rectangle

2.3. Attributes

Attributes are the units defines and describe properties and characteristics of entites Attributes are the descriptive properties by each member of an entity set for each attribute there is a set of permitted valus called domain They are represented by ellipse or oval connected to a rectangle

2.3.1. Types

  1. Single Valued

    Attributes having single value at any instance of time for an entity.

  2. Multivalued

    Attributes which can have more than one value for an entity at same time. Eg Phone NUmber Represented by double ellipse in Er diagram

  3. Simple

    Attributes which cannot divided further into sub parts

  4. Composite

    Attributes which can be further divided into sub parts as simple attributes Represented by ellipse connected to other ellipses

  5. Derived

    An attribute which can be made from another attribute Represented by dashed ellipse

  6. Stored

    Main attribute whose value is permanently stored in database

  7. Descriptive

    Attribute of relationship is called descriptive attribute Represented by ellipse connected to diamond

2.4. Relationship

It is an association between two or more entities of same/different entity set Represented by a diamond In relational model it is denoted by a foreign key or a seperate table Name - Every relation must have a unique name

2.4.1. Degree

Means number of entities set associated in the relationship set

  1. Types
    1. Binary

      Most of the relationship sets in a db system are binary

    2. Unary

      One single entity set participate in a relationship, means two entites of the same entity set are related to each other These are also called as self referential relationship set

    3. Ternary

      When three entities participate in a relationship

    4. Quaternary

      When four entities participate in a relationship

    5. N ary

      When N number of entity set are associated

2.4.2. Cardinality

The number of entities to which another entity can be associated via a relationship set

  1. Types
    1. One to One

      An entity in A is associated with at most one entity in B and entity in B is associated with at most one entity in A Represented by one arrow to another arrow

    2. One to Many

      An entity in A is associated with any number of entities in B. An entity in B however can be associated with at most one entity in A Reprented by an arrow with a line

    3. Many to One

      An entity in A is associated with any number of entities in B. An entity in B however can be associated with at most one entity in B

    4. Many to Many

      An entity in A is associated with any number of entites in B and an entity in B is associated with any number of entities in A

2.5. Participation Constraint

It defines participations of entites of an entity type in a relationship

2.5.1. Types

  1. Partial Participation

    An entity is called a strong entity set if it has a primary key all the tuples in the set are distinguishable by that key Key is represented by a underline ellipse

  2. Total Participation

    An entity set that does not process sufficient attribute to form a primary key is called a weak entity set It contains discriminator attributes which contain partial information the entity set but is not sufficient enough to identify each tuple uniquely Represented by double rectangle Discimator attribute is denoted by dashed underlined ellipse Weak relationship is represented by Double diamond

2.6. Conversion from ER to Relational Model

2.6.1. Entity Set

Convert every strong weak entity into a seperate table. In weak entity set we make it dependent onto one strong entity set (identifying or owner entity set)

2.6.2. Relationship

  • Unary - No Separate table is required, add a new column as foreign key which refer the primary key of the same table
  • 1:1 - No separate table is required, take primary key of one side and put it as foreign key on other side, Priority must be given to the side having total participation
  • 1:N - No separate table is required, modify n side by taking primary key of one side as foreign key on N side
  • M:N - Separate table is required take primary key of both table and declare their combination as a primary key of new table
  • 3:More - Take the primary key of all participating entity sets as Foreign key and declare their combinaton as primary key of the new table

2.6.3. Attributes

  • Multivalued - A separate table must be taken for all multivalued attributes, where we take primary key of the main table as foreign key and declaree combination of foreign key and multivalued attribute as primary key in the new table
  • Composite Attribute - A separate column must be taken for all simple attributes of the composite attribute

2.7. Grouped Entity sets

2.7.1. Generalization

Involves mergin two lower level entites to create a higher level entity A bottom up approach that builds complexity from simpler components Highlights similarities among lower level entity sets while hiding differences Leads to a simplifed structured data representation, aiding in database and querying process

2.7.2. Specialization

A process where a higher level entity is broken down into more specific lower level entities This top down approach delineates complexity into simpler components Acts as the converse of the generalizatoin proccess focusing on differntiating properties rather than similarities

2.7.3. Aggregation

A concept wherein relationships are abstracted to form higher-level entites, enabling a more organized representation of complex relationships

2.8. Advantages

  • Constructs used in the ER Diagram can be easily be transformed into relational tables
  • It is simple and easy to understand with minimum training

2.9. Disadvantages

  • Loss of information content
  • Limited constraints representation
  • It is overly complex for small table

3. RDBMS

Relational Database management system conceptualized by Edgar F. Codd in 1970 serves as the foundation for most database applications

3.1. Basics

Domain (set of permissible value in particular column) is a set of atomic values By atomic we mean that each value in the domain is indivisible as far as the formal relational model is concerned A common method of specifying a domain is to specify a datatype from which the data values forming the domain are drawn

3.2. Properties

  • Cells contain atomic values
  • Values in a column are of the same kind
  • Each row is unique
  • No two tables can have same name in a relational schema
  • Each column has a unique name
  • The sequence of the rows is insignificant
  • The sequence of the columns is insignificant

3.3. Problems

3.3.1. Updation Anomalies

Anomalies that cause redundant work to be done during insertion into modification of a relation and that may cause accidental loss of information during deletion from a relation

  1. Insertion Anomalies

    An indepedent piece of information cannot be recorded into a relation unless an irrelevant information must be inserted together at the same time

  2. Modification Anomalies

    The updation of a pieces of information must occur at multiple locations

  3. Deletion Anomalies

    The deletion of a piece of information unintentionally removes other information

3.4. Key

3.4.1. Super Key

Set of attributes using which we can identify each tuple uniquely is called Super Key Let X be a set of attributes in a relation R. If X+ determines the attributes of R then X is said to be Super Key of R There should be atleast one Super key in every relation

3.4.2. Candidate Key

Minimal Set of Attributes using which we can identify each tuple uniquely is called Candidate key. A super key is called candidate key if its No proper subset is a super key. Also called as Minimal Super Key There should be atleast one candidate key

3.4.3. Prime Attributes

Attributes that are member of atleast one candidate keys are called Prime attributes

3.4.4. Primary Key

One of candidate keys is selected by DBA as primary means to identify tuple is called Primary key. Primary key Attributed are not allowed to have null values. Exactly one Primary Key per Relation in RDMS. Candidate key which are not chosen as primary key are called alternative key

3.4.5. Foreign Key

A foreign key is a column or group of columns in a relational database table that refers the primary key of the same table or some other table to represent their relationship. The concept of Referential Integrity is derived from foreign key Theory.

3.4.6. Composite Key

Composite key is a key composed of more than one column sometimes it is also known as concatenated key

3.4.7. Secondary Key

Secondary key is a key used to speed up the search and retrieval contrary to primary key, a secondary key does not necessary contain unique value

3.5. Normalization

Normalization includes creating tables and establishing relationships between those tables according to the rules to protect data and make the database more flexible by elimnating redundancy and inconsistent dependency 1NF>2NF>3NF>BCNF

3.5.1. Functional Dependency

Formal tool for analysis of relational schemas If a relation R if a ( R and b ( R, then attribute or a Set of attribute a functionally derives an attribute or set of attributes of b if each a value is associated with precisely on b values For all pairs of table T1 and T2 such that If T1[a] = T2[a] Then T1[b] = T2[b]

  1. Trivial Functional Dependency

    If b is a subset of a then the functional dependency a -> b will always hold XY -> Y

  2. Attribute closure

    Attribute closure of an attribute set can be defined as set of attributs which can be functionally determined from F

  3. Partial Dependency

    When a non prime attribute is dependent only on a part of candidate key then it is called partial dependency. ( Prime -> Non Prime )

  4. Full Dependency

    When a non prime attribute is dependent on the entire candidate key it is called full dependency.

  5. Transitive Dependency

    A functional dependency from Non Prime attribute to non Prime attribute is called transitive

  6. Multivalued Dependency

    Denoted by A ->-> B means for every value of A there may exist more than one value of B

  7. Trivial Multivalued Dependency

    X -> -> Y is one where either y is a subset of X or X and Y form the whole set of attributes of the relation

3.5.2. Normal Forms

A relation with two attributes is always in BCNF A relation Schema R consist of only prime attributes then R is always in 3 NF, but may or may not be in BCNF

  1. 1 Normal Form

    1NF is the inital step of database normalization Implications of 1NF are

    • Atomic Values - Each cell in a table contains indivisible, atomic values. Meaning Relation should not contain any multivalued or composite attributes
    • Unique Columns - Each column must have a distinct name to identify the data
    • Primary Key - A table in 1NF should have a primary key that uniquely identifies the row
    • Eliminating Duplicates - Duplicate rows are removed to prevent data redundancy
  2. 2 Normal Form

    Relation R is in 2NF if

    • R should be in 1 NF
    • R should not contain any Partial Dependency ( that is every non prime attributes should be fully dependent of candidate key )
  3. 3 Normal Form

    Let R be the relational Schema, it is said to be in 3 NF

    • R should be in 2 NF

    • It must not contain any transitive dependency

      A relational schema R is said to be 3 NF if every functional dependency of form a -> b either a is super key or b is the prime attribute

  4. Boyee Codd Normal Form

    A relational schema R is said to be BCNF if every functional dependency in R form

    • a -> b
    • a must be a super key
  5. 4 Normal Form

    A relation is in 4NF if

    • It is in BCNF
    • There must not exist any non trivial multivalued dependency

    Each MVD is decomposed in seperate table, where it becomes trivial MVD

  6. 5 Normal Form

    A relational table R is said to be in 5th Normal form if

    • It is in 4NF
    • It cannot be further non loss decomposed

3.5.3. Decomposition

  1. Lossless Dependency Preserving Decomposition

    If we decompose a table r into two tables r1 and r2 because of normalization then at some later stage if want to join these tables r1 and r2 then we must get back the original table r without any change to original table Decomposition is lossy if R1 join R2 is a proper set of R Decomposition is lossy if R is a proper set of R1 join R2

    Checking

    • Union of attributes of R1 and R2 mst be equal to attribute of R
    • Insertion of attributes of R1 and R2 must not be null
    • Common Attribute must be a key for atleast one relation
  2. Dependency Preserving Decomposition

    Let relation R be decomposed in to relations R1 R2 R3 …. Rn with their respective functional dependencies set as F1, f2 f3 …. Fn then the decomposition is dependency preserving if

    • {F1 U F2 U F3 …. U Fn}+ = F+

3.5.4. Armstrong Axioms

An axiom or postulate is a statement that is taken to be true to serve as a premise or starting point for further reasoning and arguments Armostrong Axioms are a set of axioms used to infer all the functional dependencies on a relational database. The axioms are sound in generating only functional dependencies in the closure of a set of functional dependencies when applied to that set

  1. Axioms
    1. Primary Axioms
      • Reflexivity - If Y is a subset of X, then X -> Y
      • Augmentation - If X -> Y , then XZ -> YZ
      • Transitivity - If X -> Y and Y -> Z , then X -> Z
    2. Secondary Axioms
      • Union - If X -> Y and X -> Z, then X -> YZ
      • Decomposition - If X -> YZ, then X -> Y and X -> Z
      • Psuedo Transitivity - If X -> Y and WY -> Z, then WX -> Z
      • Composition - If X -> Y and Z -> W, then XZ -> YW
  2. Sound

    By Sound, We mean that given a set of functional dependencies F specified on a relation Schema R, any dependency that we can infer from F by using the primary rules of Armstrong axioms holds in every relation state r of R that satisfies the dependencies in F

  3. Complete

    By Complete, we mean that using primary rules of Armstrong axioms repeatedly to infer dependencies until no more dependencies can be inferred results in the complete set of all possible dependencies that can be inferred from F

3.5.5. Equivalence

F1+ = F2+ F1 < F2+ and F2 < F1+

3.5.6. Minimal Cover

A canoncial cover for a set of functional dependencies in a database is a minimal set of functional dependencies that is equivalent to the original set but with redundant dependencies and extraneous attributes removed. It is used in the normalization process of database design to simplify the set of functional dependencies and to find good set of relations There may be any following type of redundancy in the set of functional dependencies Complete production may be redundant One or more than one attribute may ne redundant on the right hand side of a production One or more than one attribute may ne redundant on the left hand side of a production

4. Indexing

Relational databases are based on set theory

  • In set theory the order of elements in unimportant, similarily in database tables
  • However in pratical implementation, element order in tables are often specfied
  • Various operations such as search, insertion and deletion are influenced by the order of elements in the tables
  • Elements in a table can be stored in two ways: sorted or unsorted

4.1. File Organisation

4.1.1. Ordered file organisation

  • All records in a file are ordered on some search key field
  • Here binary search is possible
  • Maintanence is costly as it requires reorganization of entire file
  • If file is unordered then no of blocks assesses required to reach correct block which contain the desired is O(log N) where n is the number of blocks

4.1.2. Unordered file organisation

  • Records are typically added at the end of the file, without following any specific order.
  • This insertion method only allows linear search, resulting in slower search times
  • Despite slow searches, maintainance is simpler
  • No reorganization of the entire file is needed
  • If file is unordered then no of block assesses required to reach correct block which contains the desired record is O(N), where n is the number of blocks

4.2. Indexing

Indexes are supplementary structure in database, aiding in swift record retrieval They enable quick data access based on particular attributes identified for indexing. This technique is similar to the index section in book. These provide secondary pathway to acccess records without changing physical position in the main file.

4.2.1. Properties

  • Indexes can be established on any relational field be it primary key or non key
  • Each attribute can have a dedicated index file, meaning multiple index files may exist for one main file
  • Index files are always organized, allowing for the utilization of binary search advantages, irrespective of the main file’s order
  • Indexing accelerates data retrieval time but also introduces space overhead for storing the index file
  • The correct block in the main file can be located with log2(number of blocks in the index file) + 1 accesses

4.2.2. Dense Vs Sparse

  • Dense index there is an entry in the index file for every search key value in the main file. This makes searching faster but requires more space to store the index to records itself. Note that it is not for every record, it is for every search key value Number of records in the main file > number of search keys in the index
  • If an index entry is created only for some records of the main file, then it is called sparse index. No. of index entries in the index file < No. of records in the main file.

4.2.3. Types

  1. Single Level
    1. Primary Index
      • Main file is always sorted according to the primary key
      • Indexing is done on primary key therefore called primary
      • Index file have two primary key and secondary anchor pointer
      • It is an example of Sparse indexing No of entries in the index file = NO of blocks acquired by the main file
    2. Clustering index

      Main file will be ordered on some non key attrbiute No of entires in ht inde file = no of values of the attribute It is an examplle of sparse as well as dense indexing

    3. Secondary index
      • Main file is order according to the attribute on which indexing is done
      • Secondary indexing can be done on key or non key attribute
      • It is an example of dense indexing
  2. MultiLevel
    1. Simple MultiLevel
    2. B tree

      A B tree of order m if non empty is an m way search tree in which

      • The root has atleast zero child nodes and at most m child nodes
      • The inertnal nodex except the root have at least ceiling(m/2) child nodes and at most m child nodes
      • All leaf nodes are on the same level
      • the number of keys in each internal node is one less than the number of child nodes these key partition the subtree of the nodes in a manner similar to that of a m -way tree
      1. Insertion
        • A B tree starts with a single root node at level 0. Once the root node is full with m -1 search key values and we attempt to insert another entry in the tree, the root node splits into two nodes at level1
        • Only the middle value is kept in the root node, and the rest of the values are split evenly between the other two nodes. When a non root node is full and a new entry is inserted into it, that nodes is split into two nodes at the same level, and the midde entry is mmoved to the parent node along with two pointers to the new split nodes
        • If the parent node is full it is also split, Slippting can propagate all the way to the root node, creating a new level it the root is split

5.1. Types

5.1.1. Procedural

Here User instructs the system to perform a sequence of operations on the database in order to compiler the desired result Means user provides both what to retrieve and how data to be retrieved

  1. Relational Algebra

    Relational Algebra like any other mathematical system provides a number of operators and use relations as operands and produce a new relation as their result Every Operator n the relational Algebra accepts relations/ tables as input arguments and returns always a single relation instance as the result without a name

    1. Basic Operators

      Fundamental operations in the relational algebra are select( Σ ), project (Π), union(U), set difference(-), Cartesian Product(X) and rename(Ρ).

      1. Project

        The project operation is a unary operation that its argument relation with certain attributes left out

    2. Derived Operators

      Operators derived from the basic operators They are join(|X|), intersection(n), Division(%) and assignment(=)

5.1.2. Non Procedural

In Non Procedural language, the user describe the desired information without giving a specific procedure for obtaining that information What data to be retrieved eg Relational Calculus, Tuple relational calculus, Domain relational calculus are declarative query lanaguage based on mathematical logic

  1. Relational Calculus

    Relational calculus is non procedural query language, where we have to define what to get and not how to get it

    1. Domain Relational Calculus

      Domain calculus differs from the tuple calculus in the type of variables used in the formulsas The variable range over a single values , we must have n of these domain variables. One for each variables An expression of the domain calculus is of the form (x1, x2, …., xn|COND(x1,x2,…..Xn+m)) where x1, x2,…., xn are domain variables that range over domains and cond is a condition of the domain relational calculus Similar to TRC we can have an unsafe expression that may genereate infinite relations

6. Transaction

A transaction is a set of logically related instructions to perform a logical unit of work A transaction should possess several properties often called ACID properties Two Operations are used in Transactions - Read, Write and Commit

6.1. ACID properties

6.1.1. A - Atomicity

A transaction is an atomic unit of processing. It should either be performed in its entirety or not performed at all Transaction is rolled back if a failure is encountered before commit

6.1.2. C - Consistency

A transaction should be consistency preserving, meaning that if it is completely executed from beginning to end without interference from other transactions, it should take the database from one consistent state to another

6.1.3. I - Isolation

A transaction should appear as though it is being executed in isolation from other transactions, even though many transactions are executing concurretly. Conversion of parallel schedule to serial schedule

6.1.4. D - Durability

The changes applied to the database by a committed transaction must persist in the database

6.2. Transaction States

6.2.1. Active

It is the inital state. Transaction remains in this state while it is executing operations

6.2.2. Partially Commited

After the final statement of a transaction has been executed, the state of transaction is partially committed as it is still possible that it may have to be aborted since the actual output may still be temporarily residing in main memory and not to disk

6.2.3. Failed

After the discovery that the transaction can no longer proceed. Such a transaction must be rolled back A transaction can fail from active or Partially Committed

6.2.4. Commited

A transaction enters committed state after successful completion of a transaction and final updation in the database

6.2.5. Aborted

A transaction is said to be in aborted state when the transaction has been rolled back and the datbase has been restored to its state prior to the start of execution.

6.2.6. Terminated

Closing of transaction Freeing of Resources

6.3. Schedule

When two or more transaction executed together or one after another then they can be bundled up into higher unit of execution called schedule

6.3.1. Serial Schedule

A serial schedule consists of sequence of instruction belonging to different transactions, where instructions belonging to one single transaction appear together. Before complete execution of one transaction another transaction cannot be started. Every serial schedule leads database into consistent state.

6.3.2. Non serial Schedule

A schedule in which sequence of instructions of a transaction appear in the same order as they appear in individual transaction but the instructions may be interleaved with instructions of different transaction i.e. Concurrent execution of transactions takes place

7. Recovery and Concurreny

7.1. Log Based Recovery

The system log or transaction log records all the changes or activites happening in the database, ensuring that transaction are durable and can be recovered in the event of a failure

7.1.1. Different types of log records

  • Ti start - This log entry indicate that the transaction Ti has started its executed
  • Ti, Xj, V1, V2 - This log entry documents a write operation. It states that transaction Ti changed the value of data item Xj from V1 to V2
  • Ti Commit - This log entry marks the successful completion of transaction Ti
  • Ti abort - This log entry denotes that the transaction Ti has been aborted, either due to an error or a rollback operation

Before any write operation modifies the database, a log record of that operation needs to created. This is to ensure that in case of a failure, the system can restore database to consistent state using the lgo records.

7.2. Deferred Vs Immediate

Deferred Database Modification Aspect Immediate database modification
Only at the commit point Write operation timing As soon as changes occur
Reduced, as changes are batched and written at once during the commit saving on intermediate I/O operations I/O operations Increased, as each change triggers immediate write operations, leading to more frequent I/O operations
Recovery Complexity Simpler, as uncommitted changes are not reflected in the database, making rollback easier in case of failure. More complex as it may requre undoing changes from uncommitted transactions that have been written to the database.
     

7.3. Shadow paging recovery technique

In the shadow paging recovery technique, the database maintains two pages tables during a transaction the current page table and the shadow page table

7.3.1. Initalization

When a transaction begins, the database creates a shadow copy of the page table. the database pages themselves are not duplicated, only the page table entries are duplicated.

7.3.2. Modification

As the transaction progresses, any changes are reflected in the current page table, while the shadow page table retains the orginial state. if a page is modified, a new copy of the page is created, and the current page table is updated to point to this new version, thereby ensuring that the shadow page table still points to the orginial unmodified page

7.3.3. Commit

Upon transaction commit, the shadow page table is discarded, and the current page table becomes the new committed state of database. The database atomically switches to using the current page table, ensuring that changes are installed all at once.

7.3.4. Recovery

In case of a system failure before the transaction commits, the database can easily recover by discarding the current page table and reverting back to the shadow page table, thereby restoring the database to its state before the transaction begins

7.4. Data Fragmentation

Data fragmentation in the context of a DBMS refers to the process of breaking down a database into smaller, manageable pieces, and distributing these pieces across a network of computers.

7.4.1. Horizontal Fragmentation

In DBMS, horizontal fragmentation divides a table into sections based on rows. Each fragment contains a subset of rows, usually segmented based on certain conditions or attributes. This can enhance performance by facilitating parallel processing and quicker data retrieval especially useful in distributed database systems

7.4.2. Vertical Fragmentation

This fragmentation type divides a database table by columns, with different sets of columns stored in seperate fragments. This approach is employed when different users require access to varied data attributes, promoting efficient data handling and minimizing data transfer times, which is advantageous in scenarios where queries only necessitate a subset of attribute

7.4.3. Hybrid Fragmentation

Hybrid or mixed fragmentationn in DBMS combines both horizontal and vertical strategies, optimizing data storage and retrieval for complex access patterns. This method can potentially offer performance improvements by utilizating the merits of both horizontal and vertical fragmentation, and is typically impmented in datbases with complex, multifaceted data access requirements

7.6. Concurrency Control

7.6.1. Time Stamping

When before entering the system a specific order is decided among the transactino, so in case of clash we can decide which one to allow and which to stop

  • With each transaction Ti in the system, we associate a unique fixed timestamp, denoted by TS(ti)
  • This timestamp is assigned by database system to a transaction at time transaction enters into the system
  • If a transaction has been assigned a timestamp TS(ti) and a new transaction Tj, enters into the system with a timestamp TS(tj), then always TS(ti) < TS(tj)

Timestamp with data item in order to assure such scheme, the protocol maintains for each data item Q two timestamp values:

  1. W-Timestamp(Q) is the largest timestamp of any transaction that executed write(Q) successfully
  2. R-Timestamp(Q) is the largest timestamp of any transaction that executed read(Q) successfully
  1. For Ti request a read(Q)

    1 if TS(t1) < W-Timestamp(Q), then Ti needs to read a value of Q that was already overwritten, Hence, the read operation is rejected, and Ti is rolled back 2 If TS(t1) >= W-Timestamp(Q), then the read operation is executed, and R-timestamp(Q) is set to the maximum of R-timestamp(Q) and TS(ti)
